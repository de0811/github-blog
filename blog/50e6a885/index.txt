                  DispatcherServlet
                </h1>
<ul>
<li>
<p>한곳에서 모든 <a href="/github-blog/blog/cc09bb8d">URI</a> 을 관리하기 위해 사용</p>
</li>
<li>
<p>가장 우선으로 모든 요청을 받는 컨트롤러</p>
<ul>
<li>스프링 MVC의 <strong>핵심</strong>이자 모든 웹 요청을 가장 먼저 받아 처리하는 <strong>프론트 컨트롤러(Front Controller)</strong><br>스프링 부트에서는 이 모든 과정이 자동 설정되어 있어 개발자가 직접 설정할 일은 거의 없음</li>
</ul>
</li>
<li>
<p><strong>과거 방식 (DispatcherServlet 없을 때)</strong></p>
<ul>
<li><code>/login</code>, <code>/articles</code> 등 각 요청마다 별도의 서블릿을 구현해야 했음</li>
<li>공통 기능(인증, 로깅 등)을 모든 서블릿에 중복으로 작성해야 해서 비효율적</li>
</ul>
</li>
<li>
<p><strong>현재 방식 (DispatcherServlet 사용)</strong></p>
<ul>
<li>모든 요청을 <code>DispatcherServlet</code>이 일단 받음</li>
<li>공통 기능을 처리한 후, 요청에 맞는 컨트롤러(핸들러)에게 작업을 위임</li>
<li>개발자는 실제 비즈니스 로직을 담은 컨트롤러만 작성하면 됨</li>
</ul>
</li>
<li>
<p><code>@WebServlet(name=??, urlPatterns = "*/*")</code> 형태로 되어 있을 컨트롤러</p>
</li>
</ul>
<ol>
<li><a href="/github-blog/blog/cc09bb8d">URI</a>를 받아서 <a href="/github-blog/blog/00e5cb0c">HandlerMapping</a>에 있는지 확인
<ol>
<li>없다면 정적 리소스에서 찾아봄</li>
<li>있다면 <a href="/github-blog/blog/6708b2d8">HandlerAdapter</a> 에게 전달</li>
</ol>
</li>
</ol>

<p><strong>요청 접수</strong></p>
<ul>

</ul>
</li>
<li>
<p><strong>핸들러 매핑 (Handler Mapping)</strong></p>
<ul>
<li>요청 <code>URI</code>에 맞는 컨트롤러(핸들러)를 찾아내도록 <a href="/github-blog/blog/00e5cb0c">HandlerMapping</a>에게 요청</li>
<li>주로 <a href="/github-blog/blog/f1963688">@RequestMapping</a>  어노테이션을 해석하는 <a href="/github-blog/blog/00e5cb0c">RequestMappingHandlerMapping</a> 이 사용됨</li>
<li><a href="/github-blog/blog/00e5cb0c">HandlerMapping</a>은 실행할 컨트롤러 메소드 정보가 담긴 <a href="#">HandlerExecutionChain</a>을 반환</li>
</ul>
</li>
<li>
<p><strong>핸들러 어댑터 조회 (Handler Adapter)</strong></p>
<ul>
<li>찾아낸 핸들러를 실행할 수 있는 <a href="/github-blog/blog/6708b2d8">HandlerAdapter</a>를 조회</li>
<li><a href="/github-blog/blog/f1963688">@RequestMapping</a> 컨트롤러는 <a href="/github-blog/blog/6708b2d8">RequestMappingHandlerAdapter</a> 가 실행함</li>
</ul>
</li>
<li>
<p><strong>핸들러 어댑터 실행 → 컨트롤러 호출</strong></p>
<ul>
<li><a href="/github-blog/blog/6708b2d8">HandlerAdapter</a>가 실제 컨트롤러의 메소드를 호출</li>
<li>이때 요청 인자(<a href="/github-blog/blog/1fd56626">@RequestParam</a>, <a href="/github-blog/blog/7d08fd5c">@RequestBody</a> 등)를 처리하여 메소드에 전달</li>
</ul>
</li>
<li>
<p><strong>ModelAndView 반환</strong></p>
<ul>
<li>컨트롤러는 비즈니스 로직 실행 후, 뷰 이름과 모델 데이터를 담은 <a href="#">ModelAndView</a> 객체를 반환</li>
<li>(<a href="/github-blog/blog/ea5774e3">@RestController</a>처럼 <a href="/github-blog/blog/3b7510b4">@ResponseBody</a>가 사용된 경우, <code>View</code>는 생략되고 <a href="/github-blog/blog/f9416c12">HttpMessageConverter</a>가 바로 응답 본문을 만듦)</li>
</ul>
</li>
<li>
<p><strong>뷰 리졸버 (View Resolver)</strong></p>
<ul>
<li><a href="#">ModelAndView</a>의 논리적인 뷰 이름(예: <code>"articles/list"</code>)을 실제 물리적인 뷰 객체로 변환</li>
<li>(예: <a href="#">ThymeleafViewResolver</a>는 해당 이름의 HTML 파일을 찾아 렌더링할 준비를 함)</li>
</ul>
</li>
<li>
<p><strong>뷰 렌더링 (View Rendering)</strong></p>
<ul>
<li><code>View</code> 객체가 모델 데이터를 바탕으로 최종 결과(HTML 등)를 렌더링하여 <a href="/github-blog/blog/f2084f0b">HttpServletResponse</a>에 담아 클라이언트에게 응답</li>
</ul>
</li>
</ol>

# DispatcherServlet
- 한곳에서 모든 [[URI]] 을 관리하기 위해 사용
- 가장 우선으로 모든 요청을 받는 컨트롤러
	- 스프링 MVC의 **핵심**이자 모든 웹 요청을 가장 먼저 받아 처리하는 **프론트 컨트롤러(Front Controller)**
  스프링 부트에서는 이 모든 과정이 자동 설정되어 있어 개발자가 직접 설정할 일은 거의 없음
- **과거 방식 (DispatcherServlet 없을 때)**
    - `/login`, `/articles` 등 각 요청마다 별도의 서블릿을 구현해야 했음
    - 공통 기능(인증, 로깅 등)을 모든 서블릿에 중복으로 작성해야 해서 비효율적
- **현재 방식 (DispatcherServlet 사용)**
    - 모든 요청을 `DispatcherServlet`이 일단 받음
    - 공통 기능을 처리한 후, 요청에 맞는 컨트롤러(핸들러)에게 작업을 위임
    - 개발자는 실제 비즈니스 로직을 담은 컨트롤러만 작성하면 됨

- `@WebServlet(name=??, urlPatterns = "*/*")` 형태로 되어 있을 컨트롤러
1. [[URI]]를 받아서 [[2.Ref(데이터 및 정보 저장)/Spring/HandlerMapping]]에 있는지 확인
	1. 없다면 정적 리소스에서 찾아봄
	2. 있다면 [[2.Ref(데이터 및 정보 저장)/Spring/HandlerAdapter]] 에게 전달

## 내부 동작 흐름
`{java}protected void doDispatch(HttpServletRequest request, HttpServletResponse response)` 함수의 동작 흐름
```plantuml
@startuml
title Spring MVC DispatcherServlet 동작 흐름

autonumber "<b>0.</b>"

actor Client
participant DispatcherServlet as DS
participant HandlerMapping as HM
participant HandlerAdapter as HA
participant Controller
participant ViewResolver as VR
participant View

Client -> DS: 1. HttpServletRequest
activate DS

DS -> HM: 2. 핸들러 조회 요청 (HandlerMapping)
activate HM
note right of HM: RequestMappingHandlerMapping이\n@RequestMapping을 해석하여\nHandlerExecutionChain을 찾음
HM --> DS: HandlerExecutionChain 반환
deactivate HM

DS -> HA: 3. 핸들러 실행 요청 (HandlerAdapter)
activate HA
HA -> Controller: 4. 컨트롤러 메소드 호출
activate Controller
note right of Controller
  @RequestParam, @RequestBody 등
  요청 인자를 처리하여 전달
end note

Controller --> HA: 5. ModelAndView 반환
deactivate Controller
note left of HA
  @RestController의 경우,
  View 정보 없이 HttpMessageConverter가
  바로 응답 본문을 처리
end note

HA --> DS: ModelAndView 전달
deactivate HA

DS -> VR: 6. 뷰 조회 요청 (ViewResolver)
activate VR
VR --> DS: View 객체 반환
deactivate VR

DS -> View: 7. 렌더링 요청
activate View
View --> Client: 최종 응답 (HTML)
deactivate View
deactivate DS

@enduml
```
1.  **요청 접수**
    - 클라이언트로부터 모든 [[2.Ref(데이터 및 정보 저장)/Spring/HttpServletRequest|HttpServletRequest]]를 받음

2.  **핸들러 매핑 (Handler Mapping)**
    - 요청 `URI`에 맞는 컨트롤러(핸들러)를 찾아내도록 [[HandlerMapping]]에게 요청
    - 주로 [[2.Ref(데이터 및 정보 저장)/Spring/Annotation/@RequestMapping|@RequestMapping]]  어노테이션을 해석하는 [[2.Ref(데이터 및 정보 저장)/Spring/HandlerMapping|RequestMappingHandlerMapping]] 이 사용됨
    - [[HandlerMapping]]은 실행할 컨트롤러 메소드 정보가 담긴 [[HandlerExecutionChain]]을 반환

3.  **핸들러 어댑터 조회 (Handler Adapter)**
    - 찾아낸 핸들러를 실행할 수 있는 [[HandlerAdapter]]를 조회
    - [[2.Ref(데이터 및 정보 저장)/Spring/Annotation/@RequestMapping|@RequestMapping]] 컨트롤러는 [[2.Ref(데이터 및 정보 저장)/Spring/HandlerAdapter|RequestMappingHandlerAdapter]] 가 실행함

4.  **핸들러 어댑터 실행 → 컨트롤러 호출**
    - [[HandlerAdapter]]가 실제 컨트롤러의 메소드를 호출
    - 이때 요청 인자([[2.Ref(데이터 및 정보 저장)/Spring/Annotation/@RequestParam|@RequestParam]], [[2.Ref(데이터 및 정보 저장)/Spring/Annotation/@RequestBody|@RequestBody]] 등)를 처리하여 메소드에 전달

5.  **ModelAndView 반환**
    - 컨트롤러는 비즈니스 로직 실행 후, 뷰 이름과 모델 데이터를 담은 [[ModelAndView]] 객체를 반환
    - ([[2.Ref(데이터 및 정보 저장)/Spring/Annotation/@RestController|@RestController]]처럼 [[2.Ref(데이터 및 정보 저장)/Spring/Annotation/@ResponseBody|@ResponseBody]]가 사용된 경우, `View`는 생략되고 [[2.Ref(데이터 및 정보 저장)/Spring/변환기/HttpMessageConverter|HttpMessageConverter]]가 바로 응답 본문을 만듦)

6.  **뷰 리졸버 (View Resolver)**
    - [[ModelAndView]]의 논리적인 뷰 이름(예: `"articles/list"`)을 실제 물리적인 뷰 객체로 변환
    - (예: [[ThymeleafViewResolver]]는 해당 이름의 HTML 파일을 찾아 렌더링할 준비를 함)

7.  **뷰 렌더링 (View Rendering)**
    - `View` 객체가 모델 데이터를 바탕으로 최종 결과(HTML 등)를 렌더링하여 [[2.Ref(데이터 및 정보 저장)/Spring/HttpServletResponse|HttpServletResponse]]에 담아 클라이언트에게 응답

## 개발자 관점에서의 예제
실제 개발 시 `DispatcherServlet`을 직접 다루진 않지만, 우리가 작성하는 컨트롤러가 이 흐름 속에서 동작함
```java
@RestController
public class HelloController {

    @GetMapping("/hello")
    public String hello(@RequestParam("name") String name) {
        return "Hello, " + name;
    }
}
```
1. 요청: 클라이언트가 /hello?name=Spring 요청
2. DispatcherServlet: 요청을 받음
3. HandlerMapping: /hello를 처리할 hello() 메소드를 찾아냄
4. HandlerAdapter: hello() 메소드를 호출하면서 @RequestParam을 해석하여 "Spring"이라는 값을 name 파라미터에 넣어줌
5. Controller: 메소드 실행 후 "Hello, Spring" 문자열 반환
6. DispatcherServlet: 반환된 문자열을 HttpMessageConverter를 통해 HTTP 응답 본문에 담아 클라이언트에게 전송

14:T10d36,<h1 style="color: #82AAFF;">
                  <span style="font-size: initial; color: #F0C292; margin-right: 0.5em;">h1</span>
                  Transaction
                </h1>
<p>하나의 과정을 안전하게 처리하도록 보장한다는 뜻<br>과정이 정상적으로 반영하는 것을 커밋(Commit) 이라 하고 실패해서 이전으로 되돌리는 것을 롤백(Rollback) 이라 함</p>
<h2 style="color: #C3E88D;">
                  <span style="font-size: initial; color: #F0C292; margin-right: 0.5em;">h2</span>
                  트랜잭션 ACID
                </h2>
<ul>
<li>원자성(Atomicity)
<ul>
<li>트랜젝션에서 실행한 작업들은 하나의 작업인 것처럼 모두 성공 하거나 실패해야함</li>
</ul>
</li>
<li>일관성(Consistency)
<ul>
<li>모든 트랜젝션은 일관성 있는 데이터베이스 상태를 유지</li>
<li>데이터베이스에서 정한 무결성 제약 조건을 항상 만족</li>
</ul>
</li>
<li>격리성(Isolation)
<ul>
<li>동시에 실행되는 트랜잭션들이 서로 영향을 미치지 않도록 격리</li>
<li>동시에 같은 데이터를 수정하지 못하도록 보호</li>
<li>격리성은 동시성과 관련된 성능 이슈로 인해 트랜잭션 격리 수준(Isolation level) 선택</li>
</ul>
</li>
<li>지속성(Durability)
<ul>
<li>트랜잭션을 성공적으로 끝내면 결과가 항상 기록</li>
<li>시스템에 문제가 발생해도 데이터베이스 로그 등을 사용해서 성공한 트랜잭션 내용을 복구 할 수 있어야함</li>
</ul>
</li>
</ul>
<h3 style="color: #C792EA;">
                  <span style="font-size: initial; color: #F0C292; margin-right: 0.5em;">h3</span>
                  트랜잭션 격리 수준(Isolation level )
                </h3>
<ul>
<li>READ UNCOMMITED (커밋되지 않은 읽기)</li>
<li>READ COMMITTED (커밋된 읽기) <font color="#00b050">일반적으로 사용</font></li>
<li>REPEATABLE READ (반복 가능한 읽기)</li>
<li>SERIALIZABLE (직렬화 가능)</li>
</ul>
<blockquote>
<p>밑으로 갈수록 높은 성능을 요구</p>
</blockquote>
<h2 style="color: #C3E88D;">
                  <span style="font-size: initial; color: #F0C292; margin-right: 0.5em;">h2</span>
                  트랜잭션의 단점
                </h2>
<h3 style="color: #C792EA;">
                  <span style="font-size: initial; color: #F0C292; margin-right: 0.5em;">h3</span>
                  트랜잭션 사용의 요구 사항
                </h3>
<ol>
<li>오토 커밋을 꺼야함</li>
<li>같은 커넥션을 전달 해야함</li>
<li>비즈니스 로직이 모두 돌고 난 뒤</li>
<li>오토 커밋을 다시 켜야함</li>
<li>커넥션을 반환해야함</li>
</ol>

# Transaction
하나의 과정을 안전하게 처리하도록 보장한다는 뜻
과정이 정상적으로 반영하는 것을 커밋(Commit) 이라 하고 실패해서 이전으로 되돌리는 것을 롤백(Rollback) 이라 함

## 트랜잭션 ACID
- 원자성(Atomicity)
	- 트랜젝션에서 실행한 작업들은 하나의 작업인 것처럼 모두 성공 하거나 실패해야함
- 일관성(Consistency)
	- 모든 트랜젝션은 일관성 있는 데이터베이스 상태를 유지
	- 데이터베이스에서 정한 무결성 제약 조건을 항상 만족
- 격리성(Isolation)
	- 동시에 실행되는 트랜잭션들이 서로 영향을 미치지 않도록 격리
	- 동시에 같은 데이터를 수정하지 못하도록 보호
	- 격리성은 동시성과 관련된 성능 이슈로 인해 트랜잭션 격리 수준(Isolation level) 선택
- 지속성(Durability)
	- 트랜잭션을 성공적으로 끝내면 결과가 항상 기록
	- 시스템에 문제가 발생해도 데이터베이스 로그 등을 사용해서 성공한 트랜잭션 내용을 복구 할 수 있어야함
### 트랜잭션 격리 수준(Isolation level )
- READ UNCOMMITED (커밋되지 않은 읽기)
- READ COMMITTED (커밋된 읽기) <font color="#00b050">일반적으로 사용</font>
- REPEATABLE READ (반복 가능한 읽기)
- SERIALIZABLE (직렬화 가능)
> 밑으로 갈수록 높은 성능을 요구

## 트랜잭션의 단점
### 트랜잭션 사용의 요구 사항
1. 오토 커밋을 꺼야함
2. 같은 커넥션을 전달 해야함
3. 비즈니스 로직이 모두 돌고 난 뒤
4. 오토 커밋을 다시 켜야함
5. 커넥션을 반환해야함

```java title:"트랜잭션 사용의 예" error:7-8
public void accountTransfer(String fromMemberId, String toMemberId, int money) throws SQLException {  
  log.info("accountTransfer fromMemberId={}, toMemberId={}, money={}", fromMemberId, toMemberId, money);  
  Connection connection = dataSource.getConnection();  
  try {  
    connection.setAutoCommit(false);  

	// 비즈니스 로직
    accountTransfer(connection, fromMemberId, toMemberId, money);  
  
    connection.commit();  
  }  
  // SQLException IllegalArgumentException 둘다 처리  
  catch (SQLException | IllegalArgumentException e) {  
    connection.rollback();  
    throw e;  
  } finally {  
    try {  
      if( connection != null ) connection.setAutoCommit(true);  
    } catch (SQLException e) {  
      log.error("error=", e);  
    } finally {  
      JdbcUtils.closeConnection(connection);  
    }  
  }  
}
```
> 비즈니스 로직은 아주 작지만 트랜잭션 기능의 소스가 너무 방대함

### 구현 기술마다 사용방법이 다른 트랜잭션
JDBC : `{java}con.setAutoCommit(false)`
JPA : `{java}transaction.begin()`

```java title:"JDBC 에서 트랜잭션 사용하는 방법" hl:4,17-29
public void accountTransferJDBC(String fromId, String toId, int money) throws SQLException {  
  Connection connection = dataSource.getConnection();  
  try {  
    connection.setAutoCommit(false);  
  
    Member from = memberRepository.findById(connection, fromId);  
    Member to = memberRepository.findById(connection, toId);  
  
    if (from.getMoney() < money) {  
      throw new IllegalArgumentException("잔액이 부족합니다.");  
    }  
  
    memberRepository.update(connection, fromId, from.getMoney() - money);  
    validation(toId);  
    memberRepository.update(connection, toId, to.getMoney() + money);  
  
    connection.commit();  
  } catch (SQLException | IllegalArgumentException e) {  
    connection.rollback();  
    throw e;  
  } finally {  
    try {  
      if( connection != null ) connection.setAutoCommit(true);  
    } catch (SQLException e) {  
      log.error("error=", e);  
    } finally {  
      JdbcUtils.closeConnection(connection);  
    }  
  }  
}
```


```java title:"JPA 에서 트랜잭션 사용하는 방법" hl:10,23-31
public void accountTransferJPA(String fromId, String toId, int money) {  
  // 엔티티 매니저 팩토리 생성  
  EntityManagerFactory emf = Persistence.createEntityManagerFactory("hello");  
  // 엔티티 매니저 생성  
  EntityManager em = emf.createEntityManager();  
  // 트랜잭션 획득  
  EntityTransaction tx = em.getTransaction();  
  try {  
    // 트랜잭션 시작  
    tx.begin();  
    // 비즈니스 로직  
    Member from = em.find(Member.class, fromId);  
    Member to = em.find(Member.class, toId);  
  
    if (from.getMoney() < money) {  
      throw new IllegalArgumentException("잔액이 부족합니다.");  
    }  
  
    from.setMoney(from.getMoney() - money);  
    to.setMoney(to.getMoney() + money);  
  
    // 트랜잭션 커밋  
    tx.commit();  
  } catch (Exception e) {  
    // 트랜잭션 롤백  
    tx.rollback();  
    throw e;  
  } finally {  
    // 엔티티 매니저 종료  
    em.close();  
  }  
}
```

### 공통적인 소스를 정리하는 방법
하나의 추상화를 만들어서 사용

`{java}PlatformTransactionManager` 이름이 원래는 `TransactionManager` 로 만들려고 했는데 다른 회사에서 `TransactionManager` 라는 이름을 선점해버려서 사용하지 못하게 되었음

```java title:"트랜잭션 공통 인터페이스"
public interface PlatformTransactionManager extends TransactionManager {  
  TransactionStatus getTransaction(@Nullable TransactionDefinition definition) throws TransactionException;  
  
  void commit(TransactionStatus status) throws TransactionException;  
  
  void rollback(TransactionStatus status) throws TransactionException;  
}
```

트랜잭션 동기화 매니저를 더 사용하기 쉽게 하기 위해 만들어 진 것이 `DataSourceUtils` 이기 때문에 이것을 이용하여 사용하여야 함
3. `{java}Connection connection = DataSourceUtils.getConnection(dataSource);` 통해서 커넥션을 받아와야 트랜잭션 동기화 매니저에서 커넥션을 관리 할 수 있음
8. close 를 하면 커낵션이 끊기게 되기에 안되고 트랜잭션 동기화 매니저를 통해 관리 되어야 하기 때문에 `{java}DataSourceUtils.releaseConnection(connection, dataSource);` 통해서 커넥션을 반환해야함
```java title:"transactionManager 사용 방법" hl:8,13-17
@Slf4j  
@RequiredArgsConstructor  
public class MemberServiceV3_1 {  
  private final PlatformTransactionManager transactionManager;  
  private final MemberRepositoryV3 memberRepository;  
  
  public void accountTransfer(String fromMemberId, String toMemberId, int money) throws SQLException, IllegalArgumentException {  
    TransactionStatus transaction = transactionManager.getTransaction(new DefaultTransactionDefinition());  
  
    try {  
      bizLogic(fromMemberId, toMemberId, money);  
  
      transactionManager.commit(transaction);  
    } catch (Exception e) {  
      transactionManager.rollback(transaction);  
      throw e;  
    }  
  }  
  
  private void bizLogic(String fromMemberId, String toMemberId, int money) throws SQLException {  
    Member from = memberRepository.findById(fromMemberId);  
    Member to = memberRepository.findById(toMemberId);  
    if (from.getMoney() < money) {  
      throw new IllegalArgumentException("잔액이 부족합니다.");  
    }  
    memberRepository.update(fromMemberId, from.getMoney() - money);  
    validation(toMemberId);  
    memberRepository.update(toMemberId, to.getMoney() + money);  
  }  
  
  private void validation(String toMemberId) throws IllegalArgumentException {  
    if ("ex".equals(toMemberId)) {  
      throw new IllegalArgumentException("toMemberId is null");  
    }  
  }  
}
```

## [[2.Ref(데이터 및 정보 저장)/Spring/공통관심사/AOP|AOP]] 이용한 트랜잭션 사용방법
> [!warning] 해당 방법은 Spring 에서 지원하는 내용이기 때문에 Spring 과 함께 동작되어야함
> 테스트할 때 @SpringBootTest 어노테이션 필요

[[2.Ref(데이터 및 정보 저장)/Spring/공통관심사/Proxy|CGLIB]] 방법을 사용하기 전에는 서비스의 비즈니스 로직과 트랜잭션 처리 로직이 함께 되어 있었지만
